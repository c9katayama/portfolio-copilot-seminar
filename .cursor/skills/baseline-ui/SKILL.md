# Baseline UI

AI生成UIの質低下を防ぐためのUIベースラインを適用する。

## 使い方

- **会話に適用**: この会話内のUI作業に、以下の制約を適用する。
- **ファイルをレビュー**: ファイルを以下の全制約で確認し、次を出力する。
  - 違反箇所（該当行・スニペットをそのまま引用）
  - なぜ重要か（1文で）
  - 具体的な修正案（コードレベル）

## スタック

- 既存のカスタム値や明示的な指定がなければ、Tailwind CSS のデフォルトを使うこと
- JavaScript でアニメーションが必要な場合は `motion/react`（旧 framer-motion）を使うこと
- Tailwind CSS での入場・マイクロアニメーションには `tw-animate-css` の利用を推奨
- クラス結合には `cn` ユーティリティ（`clsx` + `tailwind-merge`）を使うこと

## コンポーネント

- キーボード・フォーカスを扱うものは、アクセシブルなコンポーネントプリミティブ（Base UI、React Aria、Radix）を使うこと
- まずプロジェクト既存のプリミティブを使うこと
- 同一の操作面で複数のプリミティブ体系を混在させないこと
- 新規プリミティブはスタックと相性が良ければ Base UI を優先すること
- アイコンのみのボタンには `aria-label` を付けること
- 明示的に依頼されない限り、キーボード・フォーカス挙動を手書きで再実装しないこと

## インタラクション

- 破壊的・取り消し不能な操作には `AlertDialog` を使うこと
- ローディング状態には構造的なスケルトンを使うことを推奨
- `h-screen` は使わず `h-dvh` を使うこと
- 固定要素では `safe-area-inset` を考慮すること
- エラーは操作が起きた場所の近くに表示すること
- `input` や `textarea` でペーストを禁止しないこと

## アニメーション

- 明示的に依頼されない限りアニメーションを追加しないこと
- アニメーション対象はコンポジター系プロパティ（`transform`、`opacity`）に限定すること
- レイアウト系プロパティ（`width`、`height`、`top`、`left`、`margin`、`padding`）はアニメーションしないこと
- 小さな局所UI（テキスト・アイコン）以外では、描画系プロパティ（`background`、`color`）のアニメーションは避けること
- 入場時は `ease-out` を使うこと
- 操作フィードバックのアニメーションは 200ms を超えないこと
- ループアニメーションは画面外では停止させること
- `prefers-reduced-motion` を尊重すること
- 明示的に依頼されない限り独自イージング曲線を導入しないこと
- 大きな画像や全画面のアニメーションは避けること

## タイポグラフィ

- 見出しには `text-balance`、本文・段落には `text-pretty` を使うこと
- 数値データには `tabular-nums` を使うこと
- 情報密度の高いUIでは `truncate` または `line-clamp` を使うことを推奨
- 明示的に依頼されない限り `letter-spacing`（`tracking-*`）を変更しないこと

## レイアウト

- `z-index` は固定スケールを使い、任意の `z-*` は使わないこと
- 正方形要素には `w-*` + `h-*` ではなく `size-*` を使うことを推奨

## パフォーマンス

- 大きな `blur()` や `backdrop-filter` のアニメーションは行わないこと
- アニメーション中以外で `will-change` を指定しないこと
- レンダーで表現できることは `useEffect` で行わないこと

## デザイン

- 明示的に依頼されない限りグラデーションを使わないこと
- 紫や多色グラデーションは使わないこと
- グロー効果を主な操作手がかりにしないこと
- 特段の指定がなければ Tailwind CSS のデフォルトシャドウスケールを使うことを推奨
- 空状態には次のアクションを1つ明確に示すこと
- アクセントカラーは1画面あたり1種類に抑えることを推奨
- 新しい色を導入する前に、既存テーマや Tailwind CSS のカラートークンを優先すること
- DEMO など、作成途中であることがわかる文字列は入れない
- 顧客提案・金融向けでは、堅実で信頼感のあるビジュアルとし、グラフィカル要素は次の節に従うこと

## グラフィカル要素

- 数値や状態を伝える画面では、円グラフ・棒グラフ・折れ線グラフ・表などを適宜取り入れ、テキストだけに頼らないこと
- 金融・業務系で構成比や比較が主題の画面では、表だけで済ませず、棒グラフや構成比チャートなどの視覚要素を少なくとも1つ入れること
- どの視覚要素を使うかは、比較のしやすさを優先して選ぶこと。銘柄・カテゴリ間の差分比較には棒グラフを優先し、構成比の全体像を短時間で伝えたい場合にのみ円グラフを使うこと
- グラフでは色だけに頼らず、ラベル・凡例・パターン・形状などで情報が区別できるようにすること（色覚多様性への配慮）
- 画像には必ず `alt` を付けること。意味のある画像は説明的な文言、装飾用は `alt=""` でスクリーンリーダーから隠すこと
- アスペクト比が決まっている画像は `aspect-ratio` でレイアウト崩れを防ぐこと
- 一覧やスクロールで多く出る画像は `loading="lazy"` や適切な解像度で負荷を抑えること
- アイコンはプロジェクトで統一したセット（例: Heroicons, Lucide）を使い、複数ライブラリを混在させないこと
- 金融・業務系では落ち着いたトーンのビジュアルとし、信頼感を損なう過度にキャラクター的なイラストは避けること
- チャート・グラフ用ライブラリは既存スタックに合わせ、1プロジェクトで1体系に統一すること

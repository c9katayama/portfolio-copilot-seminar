# Deck Guide

## 推奨スライド構成
1. タイトル
   - 資料名
   - アプリの目的
   - 想定する利用シーン
2. アプリ概要
   - どの業務文脈で使うか
   - 何を短時間で把握できるようにするか
3. ユーザー向け機能
   - ポートフォリオ一覧と詳細表示
   - 保有銘柄・配分の確認
   - AI対話による検討支援
   - 提案書ドラフト生成
4. 操作方法
   - 1. ポートフォリオを選ぶ
   - 2. 保有内容とメモを確認する
   - 3. AIに質問する
   - 4. 提案書ドラフトを生成する
5. アーキテクチャ
   - 画面、コントローラ、サービス、リポジトリ、SQLite、AI API を番号付きで図示する
6. 技術スタック
   - Spring Boot
   - Thymeleaf / Vanilla JS
   - SQLite
   - JDBC
   - OpenAI / Anthropic
   - Cursor の rules / skills / hooks / specs
7. 必要な環境と実行前提
   - ローカル起動手順
   - API キー
   - 必要なランタイムやコマンド
8. 補足またはまとめ
   - デモで見せるポイント
   - 運用上の注意

## 内容の組み立て方
- 機能説明は利用者が得られる理解や作業短縮を中心に書く。
- 技術説明は実装の列挙だけでなく、各要素の役割が分かるようにする。
- 環境説明は必須条件と任意条件を分ける。
- AI機能は社内検討用の支援であり、規制上の助言を直接行う用途ではないことを崩さない。

## ビジュアル方針
- 信頼感のある配色を使う。
  - Primary: `#16324F`
  - Secondary: `#EAF0F6`
  - Accent: `#5B7C99`
  - Text: `#1F2933`
- 背景は濃色か淡色のどちらかに寄せ、色数を増やしすぎない。
- 本文と見出しは日本語の可読性を優先し、`Noto Sans JP`、`BIZ UDPGothic`、`Hiragino Sans`、`Yu Gothic` などの日本語フォントを使う。英字向けフォントだけで紙面を構成しない。
- タイトル、本文、注記、図表ラベルのフォントをばらけさせず、日本語フォントのウェイト差で階層を作る。
- アイコンや図形は丸みを抑え、長方形、直線、罫線、区画分けを基本にする。業務資料らしい整列を優先する。
- 角丸の四角や柔らかいカード UI を多用しない。ボックスが必要な場合は、角を立てた矩形と明確な境界線で情報を整理する。
- アーキテクチャ図や比較表は、番号付きの矩形ノードやセルを規則的に並べ、システマチックな見た目にする。
- 画面イメージを入れる場合も、説明の主役は操作意図にする。

## QA チェック
- タイトルと本文で言っていることがずれていないか。
- 機能、操作方法、アーキテクチャ、環境の 4 項目が漏れていないか。
- 用語が実装や README と一致しているか。
- 図のノードに番号が振られているか。
- `README.md` に追記が必要な場合、資料パスが反映されているか。

現在のブランチから下書きPRを作成し、レビュー方針を提案する。

## 手順

### Phase 1: 状態確認

1. `git fetch origin` でリモートを最新化
2. `git status` で未コミットの変更の有無を確認
3. `git log origin/main..HEAD --oneline` でコミット履歴を確認（main はデフォルトブランチ名に読み替える）
4. `git diff origin/main...HEAD --stat` で変更ファイル一覧を確認
5. `git diff origin/main...HEAD --shortstat` で変更行数を確認
6. 差分がなければ「デフォルトブランチと差分がありません」と表示して終了
7. `gh pr list --head $(git branch --show-current)` で既存PRの有無を確認
   - 既存PRがある場合 → Phase 3b（PR更新）へ

### Phase 2: レビュー方針の判定

変更内容から、**自己レビュー** か **他者レビュー** かを判定する。

#### 自己レビューで十分（すべて満たす場合）

| 条件 | 判定方法 |
|------|----------|
| 変更行数が少ない | 追加＋削除が 100 行以下 |
| 動作に影響しない | 変更がドキュメント・設定・ルール・スキル・コマンド等に限定 |
| テストが通る | ローカルまたはCIでテスト成功 |

**自己レビューを想定する変更の例:**

- `.cursor/` 配下（スキル・ルール・コマンド）
- `docs/` 配下
- `README.md` 等の説明文
- CI・ワークフロー設定
- テストのみの追加（既存コードに影響しない）
- 設定ファイルのフォーマット修正のみ

#### 他者レビューが必要（いずれか該当）

| 条件 | 理由 |
|------|------|
| プロダクトコードの変更 | 動作が変わる |
| API・エンドポイントの追加・変更 | インターフェースに影響 |
| DBスキーマ・データまわりの変更 | データに影響 |
| 認証・認可まわりの変更 | セキュリティに影響 |
| 変更行数が 100 行超 | 見落としのリスク |
| 複数領域にまたがる変更 | 影響範囲が広い |

### Phase 3a: PRの新規作成

1. push前に `./gradlew build` 等の検証実行を推奨（未実行の場合はユーザーに確認）
2. コミット履歴と変更内容からPRタイトルと説明文を作成
3. レビュー方針の判定結果を説明に含める
4. `gh pr create --draft` で下書きPRを作成

### Phase 3b: 既存PRの更新

既存PRがある場合:

1. PRのURLを表示
2. 説明の更新が必要なら `gh pr edit <番号> --body "..."` で更新
3. 未pushのコミットがあれば `git push` を実行

### Phase 4: 結果表示

PR作成・更新後、以下を表示する:

```
## PR作成完了

- URL: <PRのURL>
- レビュー方針: 自己レビュー / 他者レビュー
- 理由: <判定理由>
```

- **自己レビュー**: 「テストが通っていればApproveして問題ありません。不安があればReady for Reviewにし、レビュアーを指定してください」と案内
- **他者レビュー**: 「他者によるレビューを推奨します。Ready for Reviewにし、レビュアーを指定してください」と案内

## PRタイトル

- 70文字以内
- 変更の目的が分かる簡潔な文言

## PR説明の構成例

```markdown
## Summary
- 変更の概要（箇条書きで1〜3点）

## Background
- この変更が必要な理由（1〜2文）

## Review policy
- 自己レビュー / 他者レビュー推奨
- 理由: <判定理由>

## Impact
- 影響範囲（例: バックエンド / ドキュメント / なし）

## Test plan
- [ ] 確認項目1
- [ ] 確認項目2
```

## 注意

- 未コミットの変更がある場合は、先にコミットするよう促す
- 必ず `--draft` で下書きとして作成する
- レビュー方針は提案であり、最終判断はユーザーが行う
